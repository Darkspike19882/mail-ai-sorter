## Phase Scope: RPLY-01, RPLY-02, RPLY-03, RPLY-04, LOCL-02

This phase transforms the existing compose modal + Quick Reply sidebar into a unified, fast reply workflow. The core interaction model shifts from "generate draft in sidebar → transfer to modal → send" to a single inline reply surface with AI assistance and template support directly in the detail pane.

### Existing State (from codebase audit)

| What | Where | Current State |
|------|-------|---------------|
| Compose modal | `inbox.html` lines 433–470 | Full-screen modal with account/to/cc/subject/body/attachments. Triggered by `openReply()` and `openNewCompose()`. |
| Quick Reply sidebar | `inbox.html` lines 310–331 | Right-side card in detail pane with tone selector + "Entwurf erzeugen" + textarea + "In Antwort übernehmen" |
| Quick Reply API | `llm_routes.py` line 47 | `POST /api/llm/quick-reply` — accepts subject, from_addr, body, tone, language → returns {subject, reply} |
| LLM draft_reply | `llm_helper.py` line 170 | Ollama chat call with JSON output: system prompt asks for short, ready-to-send reply |
| SMTP sending | `smtp_client.py` | Full SMTP pipeline with SSL/STARTTLS, IMAP Sent save, attachment support |
| Send API | `inbox_routes.py` | `POST /api/send` — accepts account, to, cc, subject, body_text, in_reply_to, references, attachments |
| Template storage | `memory.py` | **Does NOT exist** — no `reply_templates` table. Only conversations, user_facts, email_summaries, rag_queries, email_embeddings, email_tags |
| Draft persistence | none | Compose state is ephemeral Alpine.js — closing the modal loses the draft |
| `useQuickReply()` | `inbox.html` line 1092 | Transfers Quick Reply sidebar draft into compose modal: sets compose.subject + compose.body, then calls `openReply()` |

### Design Decisions

**1. Composer placement: Inline in detail pane for replies, modal retained for new emails.**

Rationale: The current two-step flow (generate in sidebar → transfer to modal) adds friction. An inline reply area at the bottom of the detail pane eliminates the modal round-trip for replies. The existing compose modal (`showCompose`) is retained only for "Neue Email" (new outbound messages initiated from the top toolbar).

**2. Template storage: New `reply_templates` SQLite table via `memory.py`.**

Rationale: `memory.py` already manages SQLite with migrations. Adding a `reply_templates` table keeps template data local, queryable, and in the same persistence layer as everything else. No config.json bloat.

**3. AI adaptation workflow: Template insert → optional "KI anpassen" → editable draft.**

Rationale: The user inserts a template as the starting text. They can then click a single "KI anpassen" button to have Ollama rewrite the template for the current email context. The result is fully editable before sending. This avoids auto-mutation (user stays in control) while still leveraging AI.

**4. Draft auto-save: localStorage with debounce.**

Rationale: Drafts persist per `(account, folder, uid)` key in localStorage. On selecting an email with a saved draft, the inline composer pre-fills. On send or discard, the draft is cleared. No backend needed for draft storage in this phase.

**Behavior:**

- Textarea auto-grows from 4 rows to max 16 rows as the user types

- On email selection change, if a draft exists for the previous email, show a toast: "Entwurf gespeichert"

- **Ton: [current] ▼** — Tone selector dropdown (freundlich, formal, kurz, bestimmt)

- `[📎 Anhang]` — Opens file picker for attachments; attached files shown as removable chips

**Composer for "Neue Email":**
- The inline composer is reply-only. New emails still use the existing compose modal, triggered by the "Neue Email" button in the page header
- The modal (`showCompose`) is unchanged in structure but gains the template picker and tone selector toolbar

### 3.2 AI Draft Generation (RPLY-02)

**Where:** Triggered by the "✨ KI-Entwurf" button in the inline composer toolbar.

**Flow:**
1. User clicks "✨ KI-Entwurf"
2. Thin animated loading bar appears above the textarea (`bg-sky-600 animate-pulse`, 3px height)
3. `POST /api/llm/quick-reply` is called with:
   - `subject`, `from_addr`, `body` (from selectedMail)
   - `tone` (from tone selector)
   - `language`: "de"
   - **New:** `thread_context` — if the email has a thread, include the last 3 thread messages as additional context
4. Response: `{subject, reply}`
5. `reply` replaces textarea content. `subject` updates the composer subject if it was empty.
6. Loading bar disappears. Cursor placed at end of textarea.

**API enhancement needed:**
- `POST /api/llm/quick-reply` gains an optional `thread_context` field: array of `{subject, from, body_text}` for the last N thread messages
- `llm_helper.draft_reply()` prompt is extended: if `thread_context` is provided, include it in the prompt before the current email

**Loading states:**
- Button text changes to "Generiere..." with disabled state
- 30-second timeout — if exceeded, show inline error below toolbar: "KI-Antwort dauert zu lange. Ollama reagiert möglicherweise nicht."
- Ollama offline: show inline amber warning: "Ollama nicht erreichbar. KI-Funktionen benötigen einen laufenden Ollama-Dienst."

    id INTEGER PRIMARY KEY AUTOINCREMENT,

- Templates grouped by category, alphabetical within category

- Each template row shows: name (14px semibold) + first 60 chars of body (13px, truncated)
- Click a template: inserts template body into textarea at cursor position (or replaces empty textarea)

### 3.4 Template + AI Adaptation (RPLY-04)

**Where:** Triggered by the "✨ KI anpassen" button in the inline composer toolbar.

**Flow:**
1. User has inserted a template (or any text) into the textarea
2. User clicks "✨ KI anpassen"
3. `POST /api/llm/adapt-template` is called with:
   - `template_body`: current textarea content
   - `subject`, `from_addr`, `body` (from selectedMail — the email being replied to)
   - `tone`: current tone selection
   - `language`: "de"
   - `thread_context`: optional thread messages (same as 3.2)
4. Ollama rewrites the template to fit the current email context while preserving the template's intent
5. Response: `{reply: "adapted text"}`
6. Adapted text replaces textarea content. User can further edit or send.

**API endpoint needed:**

| Method | Path | Body | Response |
|--------|------|------|----------|
| POST | `/api/llm/adapt-template` | `{template_body, subject, from_addr, body, tone, language, thread_context?}` | `{reply: "adapted text"}` |

**LLM prompt for adaptation:**

```
System: Du bist ein Email-Assistent. Passe die folgende Vorlage an den Kontext der aktuellen Email an.
Behalte den Grundton und die Struktur der Vorlage bei, aber passe Details, Namen, Daten und Bezüge an die konkrete Email an.
Wenn die Vorlage nicht zur Email passt, erzeuge eine passende Antwort im gleichen Stil.
Antworte IMMER als JSON mit dem Feld: reply.

User:
Sprache: {language}
Ton: {tone}
Vorlage: {template_body}
Original-Betreff: {subject}
Absender: {from_addr}
Original-Inhalt: {body[:4000]}
{thread_context if present}
Passe die Vorlage an diese Email an.
```

**Button states:**
- Disabled when textarea is empty
- Shows "Passe an..." with spinner during generation
- Same 30-second timeout as draft generation
- Same Ollama-offline handling

## Draft Persistence

**Storage:** `localStorage` key: `mail-ai-sorter.drafts.v1`

**Structure:**
```json
{
  "account:folder:uid": {
    "body": "reply text",
    "subject": "Re: Subject",
    "to": "sender@example.com",
    "cc": "",
    "tone": "freundlich",
    "ts": "2026-04-17T14:30:00Z"
  }
}
```

**Behavior:**
- Auto-save: debounce 1000ms after last keystroke in reply textarea
- On email selection change: if previous email had draft content, save it. Then check if new email has a saved draft and restore it.
- On send: clear draft for that email
- On discard: clear draft for that email
- Maximum drafts stored: 50 (LRU eviction by timestamp)
- Drafts do NOT persist across browser sessions for sent emails — only unsent drafts

## Integration with Existing Inbox

1. **Quick Reply sidebar card REMOVED** — The right-side card (lines 310–331) is replaced by the inline composer
2. **Email-Analyse card KEPT** — The analysis card (lines 297–308) remains in the right sidebar
3. **Inline reply composer ADDED** — New section below the email body/attachments area in the detail pane
4. **Compose modal RETAINED for "Neue Email"** — The existing modal (lines 433–470) stays for new outbound messages only. Gains template picker + tone toolbar.
5. **"Antworten" button CHANGED** — Instead of opening the compose modal, it scrolls to and focuses the inline reply textarea
6. **Keyboard shortcuts EXTENDED** — New shortcuts added to the existing `document.addEventListener('keydown')` block

### Files Modified

| File | Change Type | Description |
|------|-------------|-------------|
| `templates/inbox.html` | Major | Remove Quick Reply card, add inline composer, add template picker, add template manager modal, extend Alpine.js state and methods |
| `memory.py` | Extend | Add `reply_templates` table to SCHEMA, add CRUD functions |
| `llm_helper.py` | Extend | Add `adapt_template()` method, extend `draft_reply()` to accept thread_context |
| `services/llm_service.py` | Extend | Add `adapt_template()` wrapper, extend `draft_reply()` to pass thread_context |
| `routes/llm_routes.py` | Extend | Add `POST /api/llm/adapt-template` endpoint, extend quick-reply to accept thread_context |
| New: `routes/template_routes.py` | New | Template CRUD endpoints: GET/POST/PUT/DELETE `/api/templates` |

### New Backend Table

```sql
CREATE TABLE IF NOT EXISTS reply_templates (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    body TEXT NOT NULL,
    category TEXT DEFAULT 'Allgemein',
    sort_order INTEGER DEFAULT 0,
    created_at TEXT DEFAULT (datetime('now')),
    updated_at TEXT DEFAULT (datetime('now'))
);
```
